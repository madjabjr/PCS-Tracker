# PCS Tracker

PCS Tracker is a locally hosted, secure web platform for managing military Permanent Change of Station (PCS) transitions. It consolidates the logistics of a relocation — task checklists, high-value inventory, timeline events, housing research, travel itineraries, and sensitive documents — into a single self-hosted application intended to run on a private home server or workstation rather than a public cloud service.

Access is restricted to explicitly whitelisted accounts, sensitive documents are encrypted at rest, and all data persists locally in a SQLite database that never leaves the host.

## Core Features

- **Task Management** — Inline-editable PCS checklists. Tasks support status, due dates, categories, and notes, edited directly in place without separate form screens. A baseline checklist is seeded automatically on first run.
- **Inventory** — High-value item tracking for the move, capturing item details, valuation, and optional photographs to support claims and accountability.
- **Chronological Timeline & Calendar** — A consolidated timeline of PCS milestones and dated tasks, with optional Google Calendar integration. Calendar sync is isolated to a dedicated "PCS Tracker" calendar created by the app; events generated by PCS Tracker are tagged and filtered so they never pollute or duplicate the user's primary calendar. The user's primary calendar is read in a separate, read-only path solely for conflict detection.
- **Housing** — Base waitlist tracking (position, status, application dates) alongside a rental property shortlist with rent, utility estimates, commute time, and photos. Includes BAH/OHA allowance logic to compare housing options against the relevant military housing allowance.
- **Secure Documents** — An in-memory document viewer backed by a compress-then-encrypt storage pipeline. Uploaded files are streamed through optional GZIP compression (skipped for already-compressed formats) and AES-256-GCM authenticated encryption using a key derived via HKDF-SHA256 from the application secret. Documents marked sensitive are decrypted only into browser memory for viewing — never written to disk or cache — and the viewer auto-locks after 60 seconds of inactivity, disabling right-click, print, and save shortcuts while open.

## Architecture

| Layer | Technology |
| --- | --- |
| Frontend | React 18, React Router v6, Vite, Axios |
| Backend | Python, FastAPI, SQLAlchemy (async), Authlib |
| Database | SQLite (via aiosqlite) |
| Encryption | AES-256-GCM (document pipeline), Fernet (field-level helpers) |
| Deployment | Docker / Docker Compose (optional) |

The backend exposes a JSON API under `/api`. The frontend is a single-page application that authenticates via an httpOnly JWT cookie and communicates with the backend using credentialed requests.

## Prerequisites

- Python 3.11 or later
- Node.js 18 or later (with npm)
- A Google Cloud OAuth 2.0 client (Client ID and Secret) for Google sign-in and calendar features
- Docker and Docker Compose (optional, for containerized deployment)

## Installation

### 1. Clone the repository

```bash
git clone <repository-url>
cd pcs-tracker
```

### 2. Backend dependencies

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate        # Windows (PowerShell: .venv\Scripts\Activate.ps1)
# source .venv/bin/activate    # macOS / Linux
pip install -r requirements.txt
```

### 3. Frontend dependencies

```bash
cd ../frontend
npm install
```

## Configuration & Security

### Authentication model

PCS Tracker uses a dual-authentication model:

1. **Google OAuth (primary)** — Users sign in with Google. Only email addresses present in the application whitelist are permitted; any other Google account is rejected at the callback. Whitelisted accounts may be granted an `admin` role.
2. **Local Admin (fallback)** — A single local administrator account authenticates with a username and a bcrypt-hashed password stored in the environment. This path requires no external connectivity and is intended as a fallback when Google OAuth is unavailable.

In both cases the backend issues a JWT delivered as an httpOnly cookie. The frontend sends credentialed requests and redirects to the login page on a `401` response.

### Required setup

Before running the application, a new user **must** create and populate an environment file. The backend will not start without it.

1. Copy the example file:

   ```bash
   cd backend
   copy .env.example .env        # Windows
   # cp .env.example .env         # macOS / Linux
   ```

2. Generate a bcrypt hash for the local admin password:

   ```bash
   python ../scripts/hash_password.py
   ```

3. Open `.env` and populate every value:

   | Variable | Description |
   | --- | --- |
   | `SECRET_KEY` | Secret used to sign JWTs and derive document encryption keys. Use a long, random value. |
   | `ALGORITHM` | JWT signing algorithm (default `HS256`). |
   | `ACCESS_TOKEN_EXPIRE_HOURS` | Session lifetime in hours (default `8`). |
   | `GOOGLE_CLIENT_ID` | OAuth 2.0 Client ID from the Google Cloud Console. |
   | `GOOGLE_CLIENT_SECRET` | OAuth 2.0 Client Secret from the Google Cloud Console. |
   | `ADMIN_USERNAME` | Local administrator username. |
   | `ADMIN_PASSWORD_HASH` | bcrypt hash generated by `scripts/hash_password.py`. |
   | `ADMIN_EMAIL` | Email address granted admin privileges. |
   | `FRONTEND_URL` | Frontend origin (default `http://localhost:5173`). |
   | `BACKEND_URL` | Backend origin (default `http://localhost:8000`). |
   | `DATABASE_URL` | SQLite connection string (default `sqlite+aiosqlite:///./pcs_tracker.db`). |
   | `AVIATIONSTACK_API_KEY` | Optional. Enables flight data lookups for travel itineraries. |

**Security notes:**

- `SECRET_KEY` protects both authentication tokens and the document encryption key. Changing it after documents have been uploaded will render previously encrypted documents unreadable.
- Never commit `.env` to version control.
- The OAuth redirect URI registered in the Google Cloud Console must match `BACKEND_URL/api/auth/google/callback`.

## Usage

### Running locally (development)

Start the backend from the `backend` directory:

```bash
uvicorn app.main:app --reload --port 8000
```

On startup the application creates database tables, applies migrations, and seeds the whitelist and baseline checklist. The interactive API documentation is available at `http://localhost:8000/docs`.

In a second terminal, start the frontend from the `frontend` directory:

```bash
npm run dev
```

The application is then available at `http://localhost:5173`.

### Running with Docker (optional)

With a populated `backend/.env` present, build and start both services:

```bash
docker compose up --build
```

The frontend is served on port `80` and the backend on port `8000`, with the SQLite database persisted in a named Docker volume.
